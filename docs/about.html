🌊 Overview

Northern Beaches Surf Check is an automated surf reporting system for Sydney's Northern Beaches. It runs daily at 5:45 AM via a macOS launch agent, fetches live marine and meteorological data, processes it into a detailed surf report, and deploys it as a static webpage via GitHub Pages.

This page explains how the surf report is generated — the data sources, calculations, and decisions behind every rating and recommendation.

📡 Data Sources

IMOS Satellite SST (Real-Time Water Temperature)

Instead of relying on monthly averages, the system fetches real-time sea surface temperature from the Integrated Marine Observing System (IMOS) via their public S3 bucket:

  • Product: RAMSSA L4 — Regional Australian Multi-Sensor SST Analysis (9 km resolution, gap-free)
  • Bucket: s3://imos-data/IMOS/SRS/SST/ghrsst/L4/RAMSSA/
  • Access: Public bucket, accessed via aws s3 cp --no-sign-request
  • Query: Downloads the latest NetCDF file, extracts SST for the Northern Beaches coastal edge (first ocean pixel offshore)
  • Coverage: Daily 12:00 UTC analyses, typically available by ~17:00 UTC the same day

If the S3 endpoint is unreachable or the NetCDF file cannot be parsed, the system falls back to monthly climatological averages.

Open-Meteo Marine API

Provides hourly wave and wind forecasts for the Sydney coast (-33.78, 151.30):

  • Wave Height — Significant wave height in metres (Hs)
  • Wave Period — Peak wave period in seconds
  • Wave Direction — Direction waves are coming from, in degrees from north
  • Wind Speed & Direction — 10m wind speed and direction

Tide Data — Harmonic Tide Model

Rather than relying on an external tide API (the MHL endpoint is currently unavailable), the system uses a harmonic tide model calibrated for Sydney Harbour. See the Tide section below for full details.

🌊 Effective Wave Height at Each Beach

The system computes effective height — the face height of the breaking wave at each beach — by applying two corrections to the offshore swell:

1. Headland Diffraction — Wiegel's Curves

Each beach has a primary aspect and an L/R window defined in beaches.json. Swell arriving inside the window is fully exposed (Kd = 1.0). Outside the window, energy is reduced by Wiegel diffraction:

Shadow angleKdHeight retained
0° (inside window)1.00100%
10° past0.7070%
30° past0.4040%
60° past0.22522.5%
90° past0.1515%

2. Shoaling Amplification

Longer-period waves carry more energy and amplify more as they rise up the shoreface before breaking:

PeriodShoal FactorExample
< 6s (windswell)1.0×No amplification
10s~1.2×1.0m → 1.2m
14s~1.4×1.0m → 1.4m
≥ 18s (groundswell)1.6×1.0m → 1.6m
effective_height = offshore_height × Kd × shoal_factor(period)

All heights are face height in Australian convention — the vertical trough-to-crest of the breaking wave.

🌊 Tide Height Calculation

Using a harmonic tide model based on the two principal tidal constituents for Sydney Harbour (Port Jackson / Fort Denison):

tide(t) = MSL + AM2·sin(2π·t/TM2−φM2) + AS2·sin(2π·t/TS2−φS2)
ParameterValueDescription
MSL0.9 mMean Sea Level (relative to Chart Datum)
AM20.57 mPrincipal lunar semi-diurnal amplitude
TM212.42 hLunar tidal period (12h 25m)
AS20.16 mPrincipal solar semi-diurnal amplitude
TS212.00 hSolar tidal period (exactly 12h)

The model uses an epoch-based calculation (since Jan 1, 2000) so the lunar tide advances ~50 minutes each day. Phase angles are calibrated for Sydney Harbour.

Tide Trend

The trend (rising/falling/high/low/slack) is determined by comparing the tide height now to predictions one hour before and after.

⭐ Surf Rating (0–5 ★)

The rating is computed per-beach in two stages:

1. Wave Height Score

wave_height_score = min(5, 5 × (effective_height / 1.83)0.85)

Capped at 1.83m (6 ft / overhead) — that's a perfect 5★ on height. Beyond 1.83m, height plateaus and quality factors do the differentiating. The exponent 0.85 gives a near-linear curve so small waves aren't compressed.

0.6m2 ft2.0★Waist high
1.0m3 ft3.0★Chest-shoulder
1.4m4–5 ft4.0★Head high
1.83m6 ft5.0★Overhead

2. Wave Quality Factor (multiplicative)

wave_quality = wind_quality × attack_factor × tide_factor × embayment_factor
  • Wind quality (0.15–1.0): blend of direction (offshore→onshore) and speed. A light offshore breeze scores 1.0; an onshore gale scores 0.15.
  • Attack factor (0.5–1.0): angle of swell vs beach face. 15°–45° = 1.0 (perfect peeler); 0° = 0.5 (close-out); 90° = 0.5 (extreme angle).
  • Tide factor (0.6–1.0): 1.0 at 0.5–1.5m tide height, decreasing linearly outside that range.
  • Embayment factor (0.3–1.0): how open or closed the beach is for the given swell size — see below.

Final Rating

star_rating = round(wave_height_score × wave_quality × 2) / 2

Because quality is a product (not an average), a single bad factor heavily penalises the score — just like in real surfing where onshore wind ruins a session regardless of wave height.

🏄 Board Recommendations

Board selection is based on effective face height, with adjustments for wave period and quality. On poor quality days (onshore wind, ugly tide, sloppy conditions), the effective height is adjusted downward so the system recommends more volume.

Period Adjustments

  • < 8s (windswell): Favour more volume — e.g. Shortboard → Fish/Funboard
  • ≥ 12s (groundswell) + ≥ 1.2m: Favour Step-Up for power

Quality Adjustment

When wave_quality is poor, the effective height is reduced for board selection:

adj_height = effective_height × (0.7 + wave_quality × 0.3)

Perfect conditions (quality = 1.0) → adj = effective height (no change). Terrible conditions (quality = 0.2) → adj = 76% of actual (bump up one volume category).

🌡️ Wetsuit Guide

Wetsuit recommendations are based on real-time sea surface temperature from the IMOS RAMSSA L4 satellite analysis, queried daily:

  • ≥ 22°C: Boardshorts or rash vest
  • 20–21°C: Spring suit (2 mm)
  • 17–19°C: 3/2 steamer
  • 14–16°C: 4/3 steamer
  • < 14°C: 5/4 steamer with booties, gloves, hood

Falls back to monthly climatology if the satellite data is unavailable.

Swell arriving within the window (aspect − left to aspect + right) is fully exposed. Outside it, energy is reduced by Wiegel diffraction around headlands.

The total window angle (left + right offset) also drives the Embayment Factor — see below.

🏞️ Embayment Factor

The embayment factor measures how well a beach's physical layout accommodates the incoming swell. It answers the question: "Is this beach wide enough for this wave, or will it close out?"

The Theory

Surf science defines a key ratio:

W / Lb

Where W is the beach width between headlands (proxied by the total angular window in degrees), and Lb is the wave's sub-wavelength — how much lateral room the wave rotation needs.

  • W / Lb < 2 → The beach is too narrow. Wave energy cannot spread out; the basin fills up faster than it drains → close-out.
  • W / Lb ≥ 6 → Plenty of lateral room. Sandbars fracture into deep channels; water escapes. Clean, triangular A-frame peaks peel left and right → sweet spot.

Our Implementation

We compute the embayment factor from two inputs:

W = left_offset + right_offset (degrees, from beaches.json)
Demand = effective_height × (period² / 100)
Ratio = W / Demand
embayment_factor = map(Ratio → 0.30–1.00)

The period² term is crucial — longer-period groundswell carries exponentially more energy and needs far more lateral room. A 1m wave at 6s and a 1m wave at 16s are completely different beasts in an enclosed bay.

Real-World Behaviour

BeachWindowSmall day (0.7m@8s)Big day (2.5m@14s)

This matches local knowledge: Freshwater closes out on big southerly groundswell while Long Reef and Curl Curl handle it. On small days, Freshwater's tight profile actually sharpens the waves into fun, punchy peaks.

Display

Rather than a separate column, the embayment factor is shown as a conditional emoji next to the beach name on each card:

  • ≥ 75% — no emoji (beach has enough room) ✅
  • 50–75% — ⚠️ warning (constrained, marginal quality)
  • < 50% — 🚫 danger (close-out risk, seek a wider beach)

This keeps the card layout clean while giving an instant visual read on which beaches are constrained by their layout for today's particular swell size and period.

🤖 Automation & Deployment

  1. 5:45 AM daily — macOS launch agent triggers surf_report.py
  2. Python script fetches live data from Open-Meteo APIs
  3. Queries IMOS S3 bucket for real-time SST
  4. Calculates beach-specific conditions, ratings, and recommendations
  5. Generates a static HTML page (docs/index.html)
  6. Commits changes to the local git repository
  7. Pushes to GitHub, which triggers automatic GitHub Pages rebuild
  8. Live site updates within minutes

⚠️ Known Limitations

  • Tide model uses only M2 + S2 constituents — missing N2, K1, O1 introduce up to ±0.2m error
  • Wiegel diffraction curves assume idealised headland geometry
  • Single offshore data point for all beaches — no local wind effects
  • Shoaling factor is uniform across all beaches (doesn't account for beach slope)
  • IMOS SST query requires AWS CLI
